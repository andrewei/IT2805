The changes we have made

Our original plan was to make 10 pages, but during development we realised that combining 2 of them made it easier with regards to javascript. We ended up with a total of 9 pages. Due to customer feedback, we also changed the skeleton/template page as well as the core hierarchically structure and use of input.

Core: Some feedback we recieved was that since the content may change and the customer do not necessarily have enough technical knowledge to change HTML, we decided to create inputs of static text. The inputs are txt-files that contains static text that can be edited. The corresponding Javascript-file will find the file and place the content in its rightful HTML.

Skeleton/template: Some of the things we changed in our original styling has been due to feedback, and some changes were evaluated and decided by us.

  • Background-image: We used a stock image while we waited for an image from our customer, but seeing that it is still not ready and we do not have a licence for the stock photo, we removed the image. It is still possible to put in a new picture later.
  • Game page: Due to the use of Javascript files, we did get a navbar consisting of smash logos that linked to the corresponding game-page. However, they were different files and with the placement in the hierarchy we decided to not have the navbar in the subgames, removing the need for the selected logo to glow.
  • Body width: We changed the body width from 60% to 75% of the screen, more so for text to be bigger and more distinctive of each other than size-issues
  • Navbar: The navbar does not have a selected item, nor does that item change color. Since we increased body size, the pages were large enough for content to fit on one screen and we made a title element for the pages instead to say which page it is on.
  • Banner: The logo is now on the completely left side of the screen, with the text being still to the right, but on the left side of the body.
  • Logo: The logo has been changed to a supplied image from Smash Trondheim.

Changes to main.html: The old logo is now present in top right corner. We have also decided to not show the video on this page, with the reasons being that the video is already located in a subpage of games and the space could instead by used to give critical and important messages.

Changes to events&results.html: This design was majorly changed after internally discussing the logical aspect of reported design. The new design is based on three rows. The first row is still the links between eachother. The second row is really the only row that changes between links. In recent events there is the 5 events with the newest date (including future), while in remaining events these 5 have been replaced by a single drop down menu which has all other events stored. The third row is where info on the event is stored. Name, date, info(from txt-file), link and a horizontal list of all the brackets or resultpages. This row changes dynamically based on selected event in both subsections of the page.

Changes to games.html: In every game-page, it is no direct link. It is only written which site is used to write about the game, not the specified links to the site.

Changes to contact.html: The time did not allow us to develop the form, but that may be something that can be done seperately in the future as the page has space to fit the form.

Changes to 1st required JS 'contentLoader.js': Due to new design, the selected game does not glow.

Changes to 2nd required JS 'buttonAnimate.js': We realized that this was not only difficult to develop, but also disturbing to users. We made a mouseover event with it instead.

Changes to 4nd required JS 'eventBox.js':Due to time-constraint, we had to prioritize. This js was not as importent and was sadly not developed in time.

Additional Javascript files We made several other javascript files as every html-file has their own unique js-file in addition to template.js. A lot of these js-files use the same types of function, but they are all based on what is needed for the html file. We could have seperated them into their own js-files, but it was easier this way. Functionality includes creating elements, removing elements, dynamically changing elements, reading files (both txt and xml) and working on the responses, insert data into elements and show elements on screen.

Modularity Requirement: We used one html-file (template.html) and one js-file (template.js) to create a template which all other html could use. By importing the js-file and calling 3 functions with parameters the html would have imported the entirety of template.html and also been connected to the ONLY css we have. We also managed to connect another javascript in the template and so, every html who uses template.js would also use hover.js.
